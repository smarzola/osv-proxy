# Full Repository Audit — 2026-07-11

## Audit target

- Repository: `osv-proxy`
- Audited commit: `f975830` (`v0.7.1`, `main`)
- Scope: correctness, quality, performance, architecture, design, style,
  security, tests, CI/release, and operational readiness
- Method: static review of production code, tests, configuration, workflows,
  and operator documentation; compiler/linter/test execution; locked dependency
  scan with RustSec

This is a point-in-time audit. It records findings but does not change product
behavior.

## Executive summary

The repository has a strong correctness-oriented foundation: strict
configuration deserialization, conservative policy defaults, policy rechecks
on direct artifact routes, bounded concurrency in several registry adapters,
streaming artifact delivery, generation-based SQLite updates, and broad
unit/e2e coverage. Formatting, Clippy with warnings denied, and all 251 unit
tests pass.

It is not yet ready to be treated as a hardened, high-throughput shared
service. The most urgent issue is a directly used XML dependency with two
high-severity denial-of-service advisories. The main request path also rebuilds
all HTTP clients per request, local SQLite evaluation runs synchronous work on
Tokio workers and repeats range queries per version, and most registry metadata
bodies have no size limit. Proxy-mode artifact URLs are accepted from registry
metadata without an egress policy, creating an SSRF boundary if an upstream or
its metadata is compromised. Local background sync stops at the first
ecosystem failure and then waits the full configured interval.

### Release recommendation

Do not publish another release before resolving `SEC-01`. Before operating the
proxy on a non-loopback/shared network, also resolve `SEC-02`, `SEC-03`, and
`OPS-01`, or put the service behind a trusted gateway with explicit egress and
request controls.

## High-severity findings

### SEC-01 — Locked XML parser has two high-severity DoS advisories

**Evidence:** `Cargo.toml:20`, `Cargo.lock:1057-1059`, `src/maven.rs:243`,
`src/maven.rs:432`, `src/maven.rs:508`, `src/maven.rs:516`.

The locked dependency is `quick-xml 0.38.4`. `cargo audit` reports:

- `RUSTSEC-2026-0195`: unbounded namespace-declaration allocation in
  `NsReader`, severity 7.5;
- `RUSTSEC-2026-0194`: quadratic duplicate-attribute checking, severity 7.5.

Maven POM and metadata responses are parsed with `quick_xml::de::from_str`, so
the dependency is on a reachable path. Existing 1 MiB POM and 2 MiB metadata
limits reduce maximum input size but do not remove the parser vulnerabilities,
especially the quadratic CPU case.

**Impact:** a malicious or compromised Maven upstream can cause excessive
memory use or CPU consumption on request-serving Tokio threads.

**Recommendation:** upgrade `quick-xml` to at least `0.41.0`, run the full
Maven/Gradle suite, and add `cargo audit` as a required CI and release gate.

### SEC-02 — Proxy-mode artifact delivery has an SSRF/egress-control gap

**Evidence:** `src/artifacts.rs:48-73`, `src/artifacts.rs:81-106`,
`src/npm.rs:317-359`, `src/server.rs:519-550`.

Artifact delivery accepts a metadata-derived URL as a `String` and passes it
directly to Reqwest. There is no scheme, origin, IP-range, port, or redirect
target validation. Reqwest's default redirect behavior also permits a public
URL to redirect to a private or link-local address. npm and NuGet are clear
examples of registry metadata becoming the fetch target; other adapters have
similar trust boundaries.

**Impact:** in `artifacts.behavior=proxy`, a compromised/misconfigured upstream
or attacker-controlled metadata can make the proxy send requests to internal
services or cloud metadata endpoints. Response-header filtering does not
prevent the outbound request.

**Recommendation:** centralize URL validation in artifact delivery. Permit
only `https` by default; bind each ecosystem to configured/explicit artifact
origins; reject loopback, private, link-local, multicast, and credential-bearing
URLs unless explicitly allowed; disable redirects or validate every hop after
DNS resolution. Test IPv4, IPv6, DNS rebinding, userinfo, alternate ports, and
redirect chains.

### SEC-03 — Registry metadata responses are generally unbounded

**Evidence:** `src/npm.rs:46-59`, `src/pypi.rs:52-76`, `src/nuget.rs:49-57`,
`src/go.rs:89-101`, `src/malicious.rs:680-693`. Maven and RubyGems are notable
partial exceptions with explicit body limits.

Most upstream metadata and OSV dump responses are decoded via `.json()`,
`.text()`, or `.bytes()` without enforcing a streamed byte ceiling. Limits on
parsed item counts occur only after the complete response is buffered. The
8 KiB Axum `DefaultBodyLimit` covers inbound request bodies, not upstream
responses.

**Impact:** a large legitimate package, hostile upstream, or decompression-heavy
response can consume large amounts of memory and tie up request workers. Several
simultaneous responses multiply the effect.

**Recommendation:** implement one bounded response collector used by every
metadata client, with ecosystem-specific byte and item limits, checked both
against `Content-Length` and while streaming. Disable transparent compression
or enforce a post-decompression bound. Stream OSV archives to temporary files
instead of retaining whole archives in memory.

### PERF-01 — Every HTTP request rebuilds every registry client

**Evidence:** `src/server.rs:39-50`, `src/server.rs:385-410`, and the additional
NuGet artifact client at `src/server.rs:546-547`.

The router state stores only configuration and the OSV checker. Each request
constructs npm, PyPI, Go, Cargo, artifact, NuGet, RubyGems, and Maven clients,
even though only one ecosystem route will be used. Each constructor builds a
new Reqwest client and pool.

**Impact:** connection reuse, TLS session reuse, DNS reuse, and idle pooling are
discarded. The package-install hot path pays repeated client construction cost
and opens substantially more upstream connections under load.

**Recommendation:** build immutable clients once in `AppState`. Route to an
ecosystem first, then use its shared client. Keep a single shared artifact
delivery client and add a regression test/metric for connection reuse.

### PERF-02 — Local SQLite policy evaluation blocks Tokio workers

**Evidence:** `src/malicious.rs:621-634`, `src/malicious.rs:1463-1480`.

The async `OsvChecker` implementation opens a synchronous Rusqlite connection
and runs all queries inline. The configured SQLite busy timeout is five
seconds, so lock contention can block an async runtime worker for that entire
period. Maven XML parsing and local OSV range/version parsing also perform
nontrivial synchronous CPU work on request futures.

**Impact:** a slow local query or lock stall can delay unrelated requests on
the same runtime. Concurrent installs can exhaust Tokio workers even though
the server itself is asynchronous.

**Recommendation:** move SQLite opens/queries and other material CPU parsing to
`spawn_blocking`, with an explicit concurrency semaphore. Prefer a managed
read-connection pool or per-worker read connection, and expose query duration
and busy-timeout metrics.

### PERF-03 — Local batch evaluation repeats range SQL per version and per range

**Evidence:** `src/malicious.rs:569-617`, `src/malicious.rs:1872-1880`,
`src/malicious.rs:1910-1978`, `src/malicious.rs:1981-2001`.

`check_many` groups by ecosystem, package, **and version**, then calls
`range_hits` for every unique version. `stored_ranges` reloads the same package
ranges each time and performs another SQL query for every range to load its
events. The `range_results` cache uses the same version-specific key as the
already-deduplicated group, so it provides no reuse across versions.

**Impact:** metadata containing `V` versions and `R` stored ranges causes
roughly `V * (2 + R)` queries plus repeated range parsing/comparison. This
undermines the intended low-latency benefit of local mode.

**Recommendation:** group by ecosystem/package, fetch exact-version hits for
all requested versions in one query, fetch ranges and events in one joined
query, parse them once, and evaluate all versions in memory. Add query-count
and high-version-count benchmarks.

### REL-01 — One ecosystem sync failure blocks all later ecosystems

**Evidence:** `src/server.rs:72-87`, `src/malicious.rs:697-723`,
`src/malicious.rs:737-774`.

`sync_osv` iterates ecosystems sequentially and uses `?` on each result. A
failure in npm prevents PyPI, Go, Cargo, NuGet, RubyGems, and Maven from being
attempted. The background task then sleeps the full `sync_interval` (six hours
by default) with no short retry or jitter.

**Impact:** one transient upstream error can leave unrelated datasets stale or
uninitialized for hours. On a new database, local fail-closed policy can make
those ecosystems unavailable until the next successful full pass.

**Recommendation:** isolate ecosystem sync outcomes, attempt all ecosystems,
return an aggregate report, and retry failed ecosystems with capped exponential
backoff and jitter. Ensure overlapping sync runs cannot occur and expose
per-ecosystem freshness/readiness.

## Medium-severity findings

### OPS-01 — No authentication, TLS, rate limiting, or global concurrency budget

The default loopback bind is safe for single-host use, but configuration can
bind publicly without warning. The product explicitly does not implement
authentication. Per-request fan-out has local caps (for example 16 Go info
requests and 8 NuGet registration page requests), but there is no process-wide
budget, admission control, or rate limit.

**Impact:** a shared deployment is an anonymous upstream fan-out service and
can exhaust sockets, memory, OSV quota, or registry quota.

**Recommendation:** document a hard deployment boundary: use a trusted
authenticating TLS gateway and network policy. Add global semaphores, request
deadlines, per-client limits, and a validation warning/error for non-loopback
binds unless an explicit public-service acknowledgement is configured.

### OPS-02 — Health, readiness, graceful shutdown, and actionable telemetry are absent

**Evidence:** `docs/registry-behavior.md:88-93`, `docs/observability.md:1-71`,
`src/server.rs:27-36`, `src/server.rs:76-87`.

The observability document describes desired logs and metrics, but production
code emits only ad hoc `println!`/`eprintln!` startup and sync messages. There
are no `/healthz` or `/readyz` routes, request IDs, structured decision logs,
latency metrics, per-upstream status metrics, graceful signal handling, or
drain period.

**Impact:** operators cannot distinguish a live process from one unable to
enforce policy, diagnose latency/fan-out, alert on stale datasets, or safely
remove an instance during active artifact streams.

**Recommendation:** implement structured tracing with redaction, Prometheus or
OpenTelemetry metrics, dependency-aware readiness, liveness, SIGTERM draining,
and per-ecosystem local dataset freshness.

### CI-01 — CI/release omit Clippy and dependency vulnerability gates

**Evidence:** `.github/workflows/ci.yml:93-103`; release repeats formatting and
tests but not Clippy or RustSec scanning.

Local Clippy passes with `-D warnings`, but CI does not enforce it. The current
high-severity dependency findings therefore passed both normal CI and release
validation.

**Recommendation:** require `cargo clippy --all-targets --all-features --
-D warnings`, `cargo audit`, and preferably license/source policy via
`cargo-deny` on pull requests, `main`, and tagged releases.

### CI-02 — Build and action inputs are not fully reproducible or immutable

**Evidence:** `.github/workflows/ci.yml:20-32`, `.github/workflows/ci.yml:64-72`,
and equivalent release steps.

GitHub Actions are referenced by movable major tags, Rust uses the moving
`stable` channel, and Node uses `lts/*`. The release binaries can therefore
change toolchains without a repository diff. Maven's manually downloaded
archive is correctly checksum-pinned, which is the pattern to retain.

**Recommendation:** pin actions to commit SHAs, add a committed Rust toolchain
file, pin all package-manager versions, and record the toolchain in release
provenance/SBOM output.
### CFG-01 — Configuration validation does not validate network endpoints

**Evidence:** `src/config.rs:23-70`, `src/config.rs:73-95` and the upstream URL
fields immediately following.

Validation checks policy semantics but not `server.bind`,
`server.public_base_url`, OSV API URL, or registry upstream URLs. Invalid
schemes, embedded credentials, fragments, unusable base paths, and malformed
public URLs are accepted until a request fails or a generated client URL loses
its required header.

**Recommendation:** parse bind addresses and URLs at load time into typed
values. Enforce allowed schemes, forbid userinfo/fragments, normalize trailing
paths, and test generated URLs and redirect header validity.

### ARCH-01 — Routing and policy batching are duplicated across oversized modules

`src/server.rs` is 2,649 lines and `src/malicious.rs` is 4,394 lines. Server
routing has separate production/test-oriented paths with independently
constructed dependencies. Similar “select artifacts, call `check_many`, map
cardinality/errors, evaluate decisions” logic is repeated across ecosystem
modules.

**Impact:** cross-registry invariants must be repaired repeatedly, route paths
can drift, and review surface grows faster than feature surface.

**Recommendation:** introduce an application context holding shared clients;
split each ecosystem router into a small route module; centralize batch policy
evaluation and error mapping; separate SQLite schema/sync/query code from live
OSV HTTP code. Preserve one canonical artifact-delivery path.

### PERF-04 — No metadata/decision cache or request coalescing

Every metadata request refetches registry data and reevaluates OSV policy.
Concurrent requests for the same package are not coalesced. This is documented
as future work, but it is a material production constraint rather than only an
optimization.

**Recommendation:** after correctness fixes, add a bounded cache keyed by
ecosystem/package/upstream validator plus a policy/config generation. Coalesce
identical in-flight requests. Never cache past age eligibility, policy reloads,
or local OSV generation changes.

### TEST-01 — The default e2e target hard-fails when external CLIs are absent

`cargo test --locked` includes package-manager tests that panic if Maven or
Gradle are not installed. This gives a failed local test result instead of a
clear capability skip, while CI correctly installs all tools.

**Recommendation:** keep a required fully provisioned CI job, but gate external
tool tests behind explicit features/environment checks or provide a repo-local
bootstrap command that installs/verifies exact tool versions. Report skipped
tools distinctly from failed assertions.

## Low-severity findings

### QUAL-01 — Invalid internal response status silently becomes HTTP 200

`RegistryResponse::into_http_response` uses `StatusCode::from_u16(...).unwrap_or(StatusCode::OK)`.
An internal status bug would be presented as success, masking failures.

**Recommendation:** make conversion fallible or use 500 for an impossible
status, with a test.

### QUAL-02 — Manual blocklist evaluation happens after OSV I/O

The policy engine queries OSV before checking the exact local blocklist. A
manually blocked artifact therefore incurs network/database work and can return
an OSV failure reason instead of the deterministic local rule.

**Recommendation:** evaluate unconditional local block rules before OSV. Keep
allowlist bypass semantics explicit and covered by ordering tests.

### DOC-01 — Observability requirements read as implemented behavior

`docs/observability.md` uses imperative language (“Each request should log”)
without a clear “planned/not implemented” banner, while the implementation has
none of those records or metrics.

**Recommendation:** mark the page as a target design until implementation lands
and link the production runbook to actual emitted fields only.

## Positive controls and strengths

- No `unsafe` code was found.
- Unknown YAML fields are denied throughout the public configuration model.
- Defaults fail closed for missing publish times, OSV errors, stale local data,
  malicious records, and vulnerabilities.
- Exact-version allowlist OSV bypasses require a reason.
- Direct artifact routes reconstruct the artifact and re-run policy before
  redirect/proxy delivery.
- Proxy delivery streams artifact bodies and forwards only selected request and
  response headers.
- Maven POM/metadata and RubyGems Compact Index paths have explicit body-size
  limits; Go/NuGet/RubyGems enrichment paths have per-request concurrency/item
  caps.
- SQLite bootstrap uses generation staging and atomic activation, preserving
  the prior active dataset on failed imports.
- Ecosystem-specific path/name/version validation is extensive and backed by
  focused tests, including traversal and metadata-coordinate mismatch cases.
- Release tags are checked against the Cargo version, changelog, and `main`;
  release archives include SHA-256 checksums.

## Recommended remediation sequence

### Phase 0 — Release blocker

1. Upgrade `quick-xml` to `>=0.41.0` and rerun Maven/Gradle integration tests.
2. Add RustSec and Clippy checks to CI and tagged-release validation.
3. Pin build actions/toolchains and generate an SBOM or dependency inventory.

### Phase 1 — Security and availability boundary

1. Implement artifact URL/redirect egress policy and SSRF tests.
2. Add bounded upstream response collection across every adapter.
3. Add global request/upstream concurrency controls and explicit shared-service
   deployment requirements.
4. Add health/readiness, structured logs, metrics, and graceful shutdown.

### Phase 2 — Install-path performance

1. Move all HTTP clients into shared application state.
2. Move SQLite and CPU-heavy parsing off Tokio workers.
3. Rewrite local `check_many` as package-batched SQL plus in-memory evaluation.
4. Make ecosystem sync independent, retryable, and observable.
5. Add metadata request coalescing and a correctness-aware bounded cache.

### Phase 3 — Maintainability and proof

1. Split the router, live OSV, SQLite query, and SQLite sync boundaries.
2. Centralize batch policy evaluation and error/cardinality handling.
3. Add Criterion or equivalent benchmarks for large npm/PyPI/Cargo metadata,
   Go lists, NuGet pages, local range evaluation, and concurrent installs.
4. Add fuzz/property tests for route parsing, ecosystem versions, XML/JSON
   metadata, URL validation, and OSV range events.

## Exit criteria for a hardened shared-service release

- No known high/critical advisories in locked runtime dependencies.
- Artifact proxy egress is origin/IP/redirect constrained and tested.
- Every upstream metadata/archive response has a post-decompression byte bound.
- No synchronous SQLite or material parser work runs on Tokio workers.
- HTTP clients and pools are shared across requests.
- Local batch query count scales with packages/ranges, not versions times ranges.
- A failure syncing one ecosystem does not prevent attempts for the others.
- Readiness reflects OSV mode and per-ecosystem dataset freshness.
- CI enforces format, Clippy, tests, dependency advisories, and reproducible
  toolchain inputs.
- Load tests demonstrate defined latency, memory, connection, and upstream
  request ceilings under concurrent package installs.
